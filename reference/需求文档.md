# 实验室库存管理系统需求文档

## 1. 引言

### 1.1 目的

本文档旨在详细描述多实验室库存管理系统的需求。它将作为系统设计、开发、测试和验收的基础。

### 1.2 范围

该系统面向多个实验室使用，用于管理各实验室的设备（device）和材料（parts），包括资产信息、库存、存储位置、使用状态、维护、借出、归还、跨实验室借用申请与审批等。

**部署模式：** 每个实验室独立部署自己的管理系统（独立服务器、独立数据库），通过联邦 API 协议查询其他实验室的资产信息并发起借用申请。联邦通信采用 HMAC-SHA256 签名验证确保请求来源可信和消息完整性。

**本地实验室管理：** 一个服务器实例可管理多个子实验室（如大学下多个院系），存储在 `local_laboratories` 表中。用户只能直接管理其所属实验室的数据。

**远程实验室通信：** 通过 `remote_laboratories` 表记录其他实验室的 API 地址和共享密钥，联邦 API 路由实时转发远程查询结果。

## 2. 总体描述

### 2.1 功能需求

#### 2.1.1 本地实验室管理

* 支持创建和维护多个**本地**实验室记录（存储在 `local_laboratories` 表中）。
* 支持用户、资产类别、设备、材料、库存、借用申请、维护记录与本地实验室建立归属关系。
* 支持按实验室查看、筛选、统计和导出数据。
* 支持系统管理员跨实验室管理，实验室管理员管理所属实验室。

#### 2.1.1.1 远程实验室通信

* 支持配置远程实验室的 API 地址和共享密钥（存储在 `remote_laboratories` 表中）。
* 通过 HMAC-SHA256 签名协议验证远程实验室请求的合法性和消息完整性。
* 通过远程实验室的联邦 API 查询资产目录和库存信息（实时转发，不缓存到本地数据库）。
* 联邦 API 返回的数据为脱敏的公开信息，不包含内部备注、序列号等敏感字段。
* 支持发起跨实验室借用申请，申请数据存储在本地数据库中。

#### 2.1.2 数据管理

* 设备和材料的信息记录，包括名称、型号、所属实验室等。
* 设备和材料的库存记录，包括序列号、批次、数量、状态、所属实验室等。
* 支持文档和图片的存储，如用户手册、测试报告、维护记录附件。
* 支持设备和材料信息的统一展示和查询。
* 支持设备和材料库存的统一展示和查询。
* 支持其他实验室资产基础信息的只读展示。

#### 2.1.3 批量导入和导出

* 支持 Excel、CSV 格式的批量导入。
* 批量导入时必须指定或解析所属实验室。
* 支持按实验室、资产类型、状态、时间范围导出数据。
* 数据导出功能应支持导出数据库表结构、数据、相关联的文档和图片。

#### 2.1.4 借出、归还和跨实验室借用

* 支持本实验室内部借出和归还记录。
* 支持用户对其他实验室资产发起借用申请。
* 跨实验室借用申请必须由被借实验室成员审批。
* 记录借出、归还、申请、审批的日期、负责人、申请人、审批人和状态。
* 页面提醒长期未归还和待审批的条目。

#### 2.1.5 库存管理

* 支持管理员为材料设置最低库存阈值。
* 支持按实验室配置和查看库存提醒。
* 当库存低于阈值时，系统提供明显提醒。

#### 2.1.6 检索和统计

* 支持全文检索和高级搜索功能。
* 支持按实验室、资产类别、状态、可借用性、负责人等条件检索。
* 提供统计报告，包括使用频率、借用频率、维护记录、跨实验室借用情况等。

#### 2.1.7 安全性和权限

* 用户群组包括系统管理员、实验室管理员、普通用户、访客。
* 用户权限基于角色和所属实验室共同决定。
* 系统管理员可管理所有实验室、用户和资产。
* 实验室管理员可管理所属实验室资产，并可新增、删除所属实验室的普通用户和访客用户。
* 普通用户可管理所属实验室资产，但不可管理用户。
* 用户可查看其他实验室资产的基础信息，并可申请借用。
* 详细记录操作日志，支持审计和必要的操作回退。

### 2.2 用户界面

* 清晰、简洁的布局，专业配色方案。
* 设备和材料以列表视图展示，详情通过链接访问。
* 在列表和详情页明确展示资产所属实验室。
* 提供实验室筛选器、状态筛选器和可借用性筛选器。
* 提供待审批、待归还、逾期、低库存等提醒入口。
* 响应式设计以适应不同设备。

### 2.3 性能需求

* 适应多个实验室和中小团队共同使用，保证系统稳定性和响应速度。
* 检索、统计和跨实验室资产列表应在数据量增长时保持可接受的响应时间。
* 权限过滤不得明显降低常用列表查询和搜索的响应速度。

### 2.4 设备和材料状态分类

* 设备状态包括闲置、使用中、故障、维修中、借出、待审批借用等。
* 材料状态包括可用、借出、低库存、不可用、待审批借用等。
* 跨实验室借用申请状态包括待审批、已批准、已拒绝、已取消、已借出、已归还、逾期。

### 2.5 设备和材料的维护与保养记录

* 记录维护日期、活动、负责人、所属实验室和相关文件。
* 设置定期维护提醒功能。
* 支持按实验室查看维护计划和历史记录。

### 2.6 自定义报告和数据分析

* 支持自定义报告的生成，包括选择特定的数据字段、实验室和时间范围。
* 提供数据分析功能，如趋势分析、使用频率分析、跨实验室借用分析。

### 2.7 系统维护和升级

* 根据需求变化进行不定期维护和升级。
* 升级应尽量保持现有实验室数据和权限配置兼容。

### 2.8 系统集成与扩展

* **联邦通信协议：**
  * 每个实验室独立部署，通过 HMAC-SHA256 签名 API 通信。
  * 签名包含请求方法、路径、时间戳、Nonce 和请求体哈希。
  * Nonce 去重机制防止重放攻击。
  * 时间戳验证（±900s 容差）确保请求新鲜性。
  * 生产环境应配合 HTTPS 使用，提供传输层加密。
* **联邦 API 端点：**
  * `GET /federation/info` — 实验室基本信息
  * `GET /federation/laboratories/{id}/assets` — 资产目录
  * `GET /federation/laboratories/{id}/inventory-items` — 库存查询
* **代理路由：**
  * 本地 API 转发远程查询：`GET /remote-laboratories/{id}/assets`
  * 用户通过本地 API 访问远程数据，无需直接连接远程实验室。
* 未来可考虑与其他系统集成，保留接口以支持扩展。
* API 设计应支持按实验室进行权限过滤和数据隔离。

### 2.9 备份和灾难恢复

* 每周备份策略，备份数据库、文档、图片和关键配置。
* 备份数据应能按全系统或单实验室维度恢复。
* 灾难恢复计划确保数据和系统快速恢复。

## 3. 详细功能需求

### 3.1 数据管理

#### 3.1.1 数据记录

* **实验室记录**：
  * 必须的字段：实验室名称、地址。
  * 可选字段：描述、联系方式、创建时间、更新时间。
  * 说明：实验室名称唯一。
* **资产类别记录**：
  * 必须的字段：名称、所属实验室 id。
  * 其他字段：描述。
  * 说明：同一实验室内资产类别名称唯一，不同实验室允许使用相同类别名称。
* **设备信息记录**：
  * 必须的字段：名称、型号、所属实验室 id。
  * 每个型号共有的信息：资产类别、官网页面链接、公开备注、内部备注、文档、图片。
  * 名称和型号的组合在同一实验室内唯一。
* **设备库存记录**：
  * 必须的字段：设备信息 id、所属实验室 id、序列号、状态（闲置、使用中、借出、故障、维修中）。
  * 其他信息：购买日期、存放位置、当前位置、负责人、可借用性、自定义标签、公开备注、库存内部备注、文档、图片。
  * 名称、型号、序列号的组合在同一实验室内唯一。
* **材料信息记录**：
  * 必须的字段：名称、型号、所属实验室 id。
  * 每个型号共有的信息：资产类别、官网页面链接、公开备注、内部备注、文档、图片。
  * 名称和型号的组合在同一实验室内唯一。
* **材料库存记录**：
  * 必须的字段：材料信息 id、所属实验室 id、状态（可用、借出、低库存等）、数量。
  * 其他信息：批次、购买日期、存放位置、当前位置、负责人、可借用性、自定义标签、公开备注、库存内部备注、文档、图片。
  * 批次可为空；当批次存在时，同一实验室内同一材料信息和批次组合唯一。
* 说明：所有 id 使用 UUID。
* 说明：设备和材料可以在没有库存的情况下存在信息记录，方便用户查阅信息。
* 说明：文档和图片采用本地存储，使用 URL 访问；每条记录的文档和图片都可能有多个文件，数据库中用 JSON 存储文件链接。
* 说明：其他实验室用户只能查看基础公开信息，包括名称、型号、状态、所属实验室、可借用性和公开备注；文档、图片、内部备注等敏感信息仅本实验室成员和系统管理员可见。

#### 3.1.2 新增和编辑数据

* **新增实验室记录**：
  * 仅系统管理员可新增、编辑、删除实验室记录。
  * 必填字段：实验室名称、地址。
* **新增资产类别记录**：
  * 系统管理员可为任意实验室新增资产类别。
  * 实验室管理员和普通用户只能为所属实验室新增资产类别。
  * 必填字段：名称、所属实验室 id。
* **新增设备信息记录**：
  * 必填字段：名称、型号、所属实验室 id。
  * 选填字段：资产类别 id、官网页面链接、公开备注、内部备注。
  * 可选上传：文档、图片。
* **存在对应设备信息时，新增设备库存记录**：
  * 必填字段：设备信息 id、序列号、状态（默认为闲置）、所属实验室 id。
  * 选填字段：购买日期、存放位置、当前位置、负责人、可借用性、自定义标签、公开备注、库存内部备注。
  * 可选上传：文档、图片。
* **不存在对应设备信息时，新增设备库存记录**：
  * 设备信息记录必填字段：名称、型号、所属实验室 id。
  * 设备信息记录选填字段：资产类别 id、官网页面链接、公开备注、内部备注。
  * 设备库存记录必填字段：序列号、状态（默认为闲置）。
  * 说明：在新增设备库存记录前先新增设备信息记录，设备库存记录的设备信息 id 使用新增设备信息记录的 id。
  * 说明：负责人必须是所属实验室用户或系统管理员。
* **新增材料信息记录**：
  * 必填字段：名称、型号、所属实验室 id。
  * 选填字段：资产类别 id、官网页面链接、公开备注、内部备注。
  * 可选上传：文档、图片。
* **存在对应材料信息时，新增材料库存记录**：
  * 必填字段：材料信息 id、状态（默认为可用）、数量、所属实验室 id。
  * 选填字段：批次、购买日期、存放位置、当前位置、负责人、可借用性、自定义标签、公开备注、库存内部备注。
  * 可选上传：文档、图片。
* **不存在对应材料信息时，新增材料库存记录**：
  * 材料信息记录必填字段：名称、型号、所属实验室 id。
  * 材料信息记录选填字段：资产类别 id、官网页面链接、公开备注、内部备注。
  * 材料库存记录必填字段：状态（默认为可用）、数量。
  * 材料库存记录选填字段：批次、购买日期、存放位置、当前位置、负责人、可借用性、自定义标签、公开备注、库存内部备注。
  * 说明：在新增材料库存记录前先新增材料信息记录，材料库存记录的材料信息 id 使用新增材料信息记录的 id。
* 说明：新增设备或材料库存记录时，系统应向用户展示其有权限选择的信息记录名称、型号和所属实验室。
* 说明：实验室管理员和普通用户只能新增、编辑、删除所属实验室的资产数据；系统管理员可管理所有实验室资产数据。

### 3.2 批量导入和导出

* **批量导入**：
  * 界面友好的批量导入功能，支持用户上传文件并在系统中预览。
  * 导入文件必须包含或可推断所属实验室字段。
  * 系统管理员可导入任意实验室数据。
  * 实验室管理员和普通用户只能导入所属实验室数据。
  * 错误检测和提示应包括格式错误、数据冲突、实验室权限不足、唯一性冲突等。
* **数据导出**：
  * 允许用户选择特定数据进行导出。
  * 系统管理员可跨实验室导出数据。
  * 实验室管理员和普通用户只能导出其可见范围内的数据。
  * 导出格式包括但不限于 Excel、CSV 和 PDF。
  * 敏感字段导出必须遵守权限过滤规则。

### 3.3 跨实验室借用申请、审批和归还

#### 3.3.1 跨实验室借用申请

* 用户可对其他实验室标记为可借用的资产提交借用申请。
* 必填字段：资产库存记录 id、申请人用户 id、申请人所属实验室 id、资产所属实验室 id、申请数量或设备件数、预计借出日期、预计归还日期、用途说明。
* 选填字段：备注、联系方式、附件。
* 申请状态包括：待审批、已批准、已拒绝、已取消、已借出、已归还、逾期。
* 申请提交后，资产所属实验室成员应收到待审批提醒。
* 申请人可在审批前取消申请。
* 本实验室内资产使用不建立单独借出/归还流程，通过库存状态、库存调整、分配和盘点管理。

#### 3.3.2 跨实验室借用审批

* 被借实验室的实验室管理员和普通用户均可批准或拒绝申请。
* 审批必须记录审批人、审批时间、审批结果和审批意见。
* 审批通过后，系统更新申请状态、库存状态或可用数量，并写入库存流水。
* 审批拒绝后，系统保留申请记录和拒绝原因。
* 系统管理员可查看和干预所有实验室之间的借用申请。

#### 3.3.3 跨实验室归还处理

* 归还处理需要填写归还日期（默认为系统时间）和归还状态说明。
* 系统自动更新状态为“可用”或“闲置”，或恢复材料可用数量。
* 逾期未归还的申请应在页面提醒申请人、申请人所属实验室和资产所属实验室。

### 3.4 库存管理

* **阈值设置**：
  * 允许系统管理员和实验室管理员为材料设置最低库存阈值。
  * 实验室管理员只能设置所属实验室材料阈值。
  * 当库存低于阈值时，系统提供明显提醒。
* **可借用性设置**：
  * 设备和材料库存记录应支持设置是否允许跨实验室借用。
  * 不可借用资产对其他实验室用户仅展示基础信息，不提供借用申请入口。

### 3.5 检索和统计

* **高级检索**：
  * 支持基于多个条件的高级检索，如实验室、名称、类别、状态、可借用性、负责人等。
  * 支持快速定位所需设备或材料信息。
  * 搜索结果必须根据用户权限展示不同字段。
* **统计报告**：
  * 提供设备和材料的使用频率、维护周期、库存变化、借用次数、跨实验室借用情况等统计信息。
  * 系统管理员可查看跨实验室统计。
  * 实验室管理员可查看所属实验室统计和与本实验室相关的跨实验室借用统计。
  * 数据可视化展示，如图表和图形。

### 3.6 安全性和权限

#### 3.6.1 用户账号

* **用户群组**：
  * 必须的字段：群组名。
  * 可选字段：描述。
  * 默认群组为系统管理员、实验室管理员、普通用户、访客。
* **用户账号记录**：
  * 必须的字段：用户名、密码、群组 id。
  * 实验室管理员和普通用户必须绑定所属实验室 id。
  * 访客用户可绑定实验室 id；绑定后可按所属实验室规则查看本实验室基础信息。
  * 系统管理员不强制绑定实验室 id。
  * 可选字段：邮箱、创建时间、最近登录时间。
  * 密码以哈希存储。
  * 创建时间、最近登录时间由系统自动记录，以时间戳形式存储。

#### 3.6.2 用户管理权限

* 系统管理员：
  * 可新增、编辑、删除所有实验室。
  * 可新增、编辑、删除所有用户。
  * 可设置用户群组和所属实验室。
  * 可管理所有实验室的资产、借用申请、审批和日志。
* 实验室管理员：
  * 可新增、删除所属实验室的普通用户和访客用户。
  * 可编辑所属实验室普通用户和访客用户的基础信息。
  * 不可新增、删除系统管理员。
  * 不可新增、删除其他实验室用户。
  * 不可将用户提升为系统管理员。
* 普通用户：
  * 不可管理用户。
  * 可新增、编辑、删除所属实验室的资产信息和库存记录。
  * 可查看其他实验室资产基础信息并发起借用申请。
* 访客：
  * 仅具有只读权限。
  * 可见范围由系统管理员或实验室管理员配置。

#### 3.6.3 资产权限

* 系统管理员可查看和修改所有实验室资产。
* 实验室管理员和普通用户可查看、修改、删除所属实验室资产。
* 实验室管理员和普通用户可查看其他实验室资产基础信息。
* 实验室管理员和普通用户不可直接修改其他实验室资产。
* 跨实验室借用必须通过申请和审批流程完成。

#### 3.6.4 操作日志

* 系统自动记录所有用户的操作历史，包括时间、操作类型、操作内容和操作者。
* 实验室、用户、资产、借用申请、审批、借出、归还等关键操作必须写入操作日志。
* 日志应记录操作者、操作者所属实验室、目标实验室、资源类型、资源 id、操作类型、变更前后摘要和时间。
* 系统管理员可查看全部日志。
* 实验室管理员可查看所属实验室相关日志。

### 3.7 设备、材料和申请状态分类

* **设备状态**：
  * 闲置、使用中、借出、故障、维修中、待审批借用。
* **材料状态**：
  * 可用、借出、低库存、不可用、待审批借用。
* **跨实验室借用申请状态**：
  * 待审批、已批准、已拒绝、已取消、已借出、已归还、逾期。
* **状态转换规则**：
  * 普通资产状态变更只能由资产所属实验室成员或系统管理员执行。
  * 跨实验室借用申请批准后，资产状态或数量应自动联动。
  * 申请归还完成后，资产状态或数量应恢复。

### 3.8 设备和材料的维护与保养

* **维护记录**：
  * 记录每次维护的日期、内容、负责人、所属实验室和相关文件。
  * 关联相应维护文档。
  * 内部维护备注仅所属实验室成员和系统管理员可见。
* **维护提醒**：
  * 根据设备使用情况和厂家推荐，设置定期维护提醒。
  * 实验室管理员可管理所属实验室维护提醒。

### 3.9 多实验室可见性和数据隔离

* 系统应区分公开字段和内部字段。
* 公开字段包括：名称、型号、资产类别、状态、所属实验室、可借用性、公开备注。
* 内部字段包括：内部备注、采购信息、详细库存备注、非公开附件等。
* 本实验室成员和系统管理员可查看本实验室资产完整信息。
* 其他实验室用户默认只能查看公开字段。
* 所有写操作必须校验操作者角色和实验室归属。

### 3.10 用户界面和交互

* **响应式设计**：
  * 确保界面在不同设备（如台式电脑、平板电脑、智能手机）上均有良好展现。
  * 界面元素适应不同屏幕尺寸，保持用户操作的直观性和便捷性。
* **导航和布局**：
  * 清晰的导航结构，方便用户快速找到所需功能。
  * 重要信息（如警告、提醒、待审批申请）显著展示。
  * 支持按实验室切换或筛选资产视图。
  * 资产详情页应明确区分公开信息和内部信息。

### 3.11 系统集成与扩展

* **API 和接口**：
  * 设计灵活的 API，便于未来与其他系统（如财务系统、库存管理系统）集成。
  * API 必须支持实验室维度的权限过滤。
  * 预留接口以支持未来功能扩展和技术升级。

### 3.12 报告和文档

* **系统文档**：
  * 提供系统使用手册，详细描述功能和操作步骤。
  * 文档应包括常见问题解答和故障排除指南。
  * 文档应说明多实验室权限、跨实验室借用申请和审批流程。
* **开发文档**：
  * 编写开发文档，记录系统架构、代码结构和开发规范。
  * 文档应便于新开发人员快速理解和参与项目。

### 3.13 备份和灾难恢复

* **备份策略**：
  * 明确备份的数据范围，包括数据库、文档和其他重要数据。
  * 支持按全系统或单实验室维度检查备份完整性。
  * 定期检查备份的完整性和恢复能力。
* **灾难恢复演练**：
  * 定期进行灾难恢复演练，确保恢复流程的有效性和团队的熟练度。

### 3.14 系统性能和可靠性

* **性能要求**：
  * 系统应能够快速响应用户请求，尤其是数据检索、权限过滤和报告生成。
  * 处理大量数据时，系统应保持稳定，避免长时间延迟或崩溃。
* **可靠性**：
  * 系统应具备高可靠性，减少故障和意外中断。
  * 应对常见故障提供快速恢复能力。

### 3.15 维护和升级

* **维护计划**：
  * 系统将根据需求变化进行不定期维护，以确保其稳定和高效运行。
  * 维护内容可能包括系统优化、功能更新和安全性提升。
* **升级策略**：
  * 系统升级将根据实际需求和技术进步灵活安排。
  * 升级过程中，应尽量减少对现有用户操作的影响，并提前进行通知。

### 3.16 用户反馈和支持

* **用户反馈**：
  * 目前不设置专门的用户反馈机制，但未来可能根据需要加入此功能。
  * 初始阶段，用户可通过电子邮件或直接联系提供反馈和建议。
* **技术支持**：
  * 在初始阶段，不提供系统范围内的正式用户支持和培训。
  * 未来可根据用户需求和系统发展考虑增加技术支持服务。

### 3.17 数据安全和隐私

* **数据保护**：
  * 实施必要的数据加密和保护措施，确保所有存储和传输的数据安全，尤其是敏感信息。
  * 设置访问控制，确保只有授权用户可以访问敏感和关键数据。
  * 跨实验室可见数据必须严格限定为公开字段。
  * 联邦通信使用 HMAC-SHA256 签名验证请求来源和消息完整性，配合 HTTPS 实现端到端加密传输。
  * 共享密钥仅存储在本地数据库中，不通过网络传输。
  * Nonce 去重机制防止重放攻击，时间戳验证确保请求新鲜性。

### 3.18 灾难恢复和业务连续性

* **灾难恢复计划**：
  * 设计灾难恢复策略，以应对可能的系统故障或数据丢失，包括定期的数据备份和恢复测试。
  * 确保关键业务流程的快速恢复能力，以减少任何潜在的运营中断。

### 3.19 系统监控和报告

* **性能监控**：
  * 实施系统性能监控，以确保高效运行，并及时发现及响应任何性能问题。
  * 定期生成系统运行报告，包括关键性能指标和系统健康状况。
* **日志和审计**：
  * 系统应记录详细的操作日志，便于问题追踪和安全审计。
  * 定期审查系统日志，确保系统安全和有效运行。

### 3.20 用户培训和支持

* 虽然当前不需要提供正式的用户培训和支持，但系统应设计得直观易用，以确保用户能够快速上手。
* 提供在线帮助文档，包括常见问题解答和操作指南，帮助用户解决基本问题。
* 帮助文档应包含多实验室协作、资产可见性、借用申请审批和权限边界说明。

### 3.21 系统扩展性

* 系统应设计为可扩展的，能够适应未来可能的需求变更和新功能添加。
* 保留接口和模块化设计，便于未来集成新的技术或第三方服务。
* 多实验室权限模型应支持未来扩展到部门、项目组或课题组等更细粒度组织结构。

### 3.22 质量保证和测试

* 实施全面的质量保证流程，包括单元测试、集成测试和系统测试，确保系统可靠性和性能。
* 权限、实验室归属、跨实验室借用审批、敏感字段可见性必须作为重点测试场景。
* 定期进行性能评估和安全漏洞扫描，以确保系统持续稳定。
